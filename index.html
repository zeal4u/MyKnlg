实验内容

要求输出一个文件，给每个用户推荐最多十个好友，格式为
0 12,23,31,… 说明给用户推荐的好友有12,23,31等，他们的关系强度递减

实验思路及代码

这只是一次实验，用了三次MapReduce

  • 第一次Map将输入文件整合为key:"id1(2)" value:"id1 id2"这样的“朋友对”格式切割,其中都以\t间隔，且第二个id永小于第三。
    比如 某一行的数据为：0 1,2,3 那么会变成
key value
0 0 1
0 0 2
0 0 3

… …

具体可见job1Mapper。
然后数据被按照key分给各个Reducer, 由 job1Reducer处理，将朋友对分为一度朋友 “deg1friend”（已经是朋友）和伪二度朋友“deg2friend”（可能不是朋友），然后输出。

  • 第二次MapReduce是将上一次的结果作为输入。Map并没做什么，只是将第一个输出作为输入，并按朋友对分发，给Reducer进行“支持数”统计。
    其中出现的所有的朋友对都被定义为“deg2friend”的才会被视为真正的二度朋友，因为没有之前的任何一个用户定义为一度朋友。具体见job2Mapper和job2Reducer。
    这样我们得到了所有二度朋友对的支持数统计。

  • 第三MapReduce其实就是对每一个用户所在的朋友对的支持数进行排序，选取前十个进行推荐。

CPU Schedule Strategies

发表于 2016-05-30   |   分类于 Operating System ， CPU   |  

CPU调度总结

单队列

Criteria
Quantum \ Priority Criteria Arrival Time Cost Time Behavior
Finite Round-Robin
Infinite First-Come-First-Serve Shortest-Job-First Priority Schedule

note：

  • SJF 有理论最小等待时间，但是因为任务运行需要预测，所以实际上是无法达到最优
  • RR 时间片不能设置太小，否则系统开销（任务的交接（switch）和交换（swap））过大；也不能设置太大，导致响应时间过长，失去了RR的优势
  • Priority Scheduling 可能导致Starvation，可以通过Aging方法解决

多级队列和多级反馈队列

多级队列

将进程（任务）按类型分为两个或多个队列进行调度，队列之间是相互独立的

  • foreground(interactive) RR - Round Robin
  • background(batch) FCFS

有两种模式

  • Fixed Priority
    先完成所有foreground然后再完成background，虽然响应速度快，可能导致Starvation

  • Time Slice 80% -> Foreground 20%-> Background

多级反馈队列

与上面不同之处在于，进程（任务）可在队列之间移动；且高优先级进程完成后，低优先级进程才能运行

比如： 有三个队列策略分别为

  • Q0 6ms的RR，如果没有在6ms内完成则进入Q1
  • Q1 16ms的RR，没有完成抢占式进入Q2
  • Q2 FCFS，留在本队列直到完成

捉弄人的空格

今天做linux的shell编程实验发现，shell 中 if 判断句真是一个挺要注意的部分。

比如

if [ $1 = "n" ];then
    asklist = n
else
    asklist = y
fi

‘if’和’[‘中括号之间、中括号和表达式之间要有空格，这是众所周知的。

今天我发现如果表达式是一个等式判断，那么‘=’两端也需要空格，调试了半天，着实头疼。如果写成了$1="n"脚本是不会正常工作的，反而会被识别为赋值，从而导致错误的流程。

shell中空格问题只是编程世界的一隅，在旧版本的c++中使用模板时也需要添加额外的空格，譬如 vector<vector<int> >，好在现在已经不需要这种“莫名的规矩”了。某些编译器可能仍需要以老式的声明方式来处理，需要注意。

无独有偶，我昨天在配置博客时，也遇到了配置文件属性配置了却没有生效的问题，原因也是属性和之间虽然有‘：’相隔，但还是需要添加一个额外的空格才能使之生效。

deploy:
  type: git #此处有一空格

同样的为了让我这个博客的主题生效，所有md标题的’#’和内容之间也需要加一个空格，然而这不是一般的markdown所必须的。
